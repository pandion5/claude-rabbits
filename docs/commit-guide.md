# 커밋 메시지 가이드

이 문서는 `Rabbits` 리포에서 커밋 메시지를 작성할 때 따르는 규칙을 정의한다.

## 1. 왜 좋은 커밋 메시지가 중요한가

이 리포는 스펙 → 계획 → 태스크 단위 구현이 커밋 단위로 이어지는 문서/스킬 중심 프로젝트다.
코드 diff만 봐서는 "왜 이렇게 바꿨는지"가 드러나지 않는 경우가 많고, 나중에
`git log`나 `git blame`으로 특정 결정의 배경을 추적할 일이 잦다. 좋은 커밋 메시지는:

- 리뷰어(사람 또는 리뷰 서브에이전트)가 diff를 열기 전에 변경의 의도를 파악하게 한다.
- 이후 스펙을 수정하거나 되돌릴 때 "이 커밋이 왜 필요했는지"를 즉시 알려준다.
- `git log --oneline`만으로 프로젝트 진행 흐름(스펙 → 매니페스트 → 스킬 → 아키타입 → 루브릭 → README)을
  읽을 수 있게 한다.

나쁜 커밋 메시지는 diff를 다시 열어 추론하게 만들고, 그 비용은 작성자 본인을 포함해
이 히스토리를 다시 읽는 모두가 반복해서 치르게 된다.

## 2. 이 리포의 형식

### 제목 라인

```
type: 한국어 요약
type: 한국어 요약 — 상세 보충
```

- `type`은 영문 소문자, 콜론(`:`) 뒤에 공백 하나, 그다음 한국어 요약이 온다.
- 상세 보충이 필요하면 요약 뒤에 ` — `(스페이스 + 줄표 + 스페이스)로 이어 붙인다.
  줄표는 하이픈(`-`)이 아니라 em dash(`—`)를 쓴다.
- 실제 사용 예:
  - `feat: 검토 루브릭·에스컬레이션 사다리 추가`
  - `docs: README 추가 — 설치·사용법·스모크 체크리스트`
  - `docs: 스펙 §7 감독 메커니즘 정정 — 서브에이전트 제어 제약 반영`

### 본문(선택)

- 제목과 본문 사이는 빈 줄 하나로 구분한다.
- 여러 변경 사항을 나열할 때는 `- `로 시작하는 불릿 리스트를 쓴다.
- 짧은 맥락 설명은 불릿 없이 문단으로 써도 된다.

### 푸터(선택, 자동 생성 태그 포함)

- `Co-Authored-By: <이름> <이메일>`, `Claude-Session: <URL>` 같은 태그는 Claude Code가
  세션 정보를 남기기 위해 자동으로 붙이는 푸터다. 도구가 추가한 경우 임의로 지우지 않는다.
- 사람이 직접 추가하는 푸터가 필요하면(예: 이슈 참조, 브레이킹 체인지 고지) 본문과
  빈 줄로 구분한 별도 줄에 `관련: #이슈번호`, `BREAKING CHANGE: 설명` 형식으로 남긴다.
- 자동 태그와 사람이 쓴 푸터를 섞을 경우, 사람이 쓴 푸터를 위쪽에 두고 자동 태그는
  맨 마지막에 둔다.

## 3. type 목록과 용도

이 리포는 Conventional Commits 계열 접두어를 쓰되, 문서/스킬 중심 프로젝트 특성에 맞춰
아래 type을 사용한다.

| type | 용도 | 이 리포 예시 상황 |
|---|---|---|
| `feat` | 새 스킬·아키타입·매니페스트·루브릭 등 새로운 기능/산출물 추가 | `feat: run 스킬 추가 — 6단계 자율 오케스트레이션 프로토콜` |
| `fix` | 기존 스펙·스킬·스크립트의 오류·버그 수정 (동작이 잘못되어 있었던 것을 바로잡음) | 워커 파견 조건이 잘못 기술돼 있던 것을 정정하는 경우 |
| `docs` | 스펙·계획·README 등 문서 신설 또는 정정 (동작 변경 없음) | `docs: README 추가 — 설치·사용법·스모크 체크리스트` |
| `refactor` | 동작은 그대로 두고 문서/스킬 구조·표현을 재구성 (섹션 재배치, 중복 제거 등) | 스펙 섹션 순서를 재배치하되 내용은 동일하게 유지하는 경우 |
| `test` | 스모크 체크리스트·검증 시나리오·검증 스크립트 추가·수정 | `scripts/` 아래 스모크 테스트 스크립트를 추가하는 경우 |
| `chore` | 위 어디에도 속하지 않는 잡무성 변경 (설정 파일, gitignore, 정리 작업 등) | `.gitignore` 정리, 파일 이동 등 |

- 리포 히스토리에는 현재 `feat`, `docs`가 주로 쓰였지만, 위 표의 다른 type도 해당 상황이면
  그대로 사용한다. 새 type을 임의로 만들지 말고 이 표 안에서 가장 가까운 것을 고른다.
- 하나의 커밋이 여러 type에 걸치면(예: 기능 추가 + 문서 갱신) 더 비중이 큰 변경을 기준으로
  type을 정하고, 필요하면 커밋을 나눈다.

## 4. 제목 규칙

- **길이**: `type: `를 포함해 50자 내외를 목표로 하고, 72자를 넘기지 않는다.
  em dash 뒤 상세 보충까지 포함한 길이다. 넘어갈 것 같으면 상세 보충을 줄이거나
  본문으로 옮긴다.
- **어미**: 명사형 종결(`~추가`, `~정정`, `~반영`, `~수정`)로 통일한다.
  `~합니다`, `~했음`, `~함`처럼 문장체·구어체 어미를 섞지 않는다.
- **마침표 금지**: 제목 끝에 마침표(`.`)를 붙이지 않는다.
- **구두점**: 여러 항목을 나열할 때는 가운뎃점(`·`)을 쓴다. 쉼표(`,`)를 남발하지 않는다.
  예: `검토 루브릭·에스컬레이션 사다리 추가`.
- **동사 위치**: 한국어 어순에 맞춰 목적어(무엇을) 뒤에 동사(어떻게)를 둔다.
  `추가 스킬 run` 같은 영어 어순 직역을 쓰지 않는다.
- **type과 요약의 관계**: 요약만 읽어도 무엇이 바뀌었는지 알 수 있어야 한다.
  `docs: 수정`처럼 대상이 빠진 제목은 금지한다.

## 5. 본문 작성 규칙

- **무엇(what)보다 왜(why)**: diff를 보면 무엇이 바뀌었는지는 이미 알 수 있다.
  본문에는 왜 그렇게 바꿨는지, 어떤 문제를 해결했는지, 어떤 대안을 버렸는지를 쓴다.
  단순 나열이 필요하면 "무엇을 바꿨는지"를 최소한으로 요약하고 그 아래에 이유를 붙인다.
- **줄바꿈 폭**: 본문 각 줄은 대략 72~80자에서 줄바꿈한다. 터미널이나 `git log`에서
  가로 스크롤 없이 읽히게 하기 위함이다. 불릿 항목이 길어지면 다음 줄에서
  들여쓰기(공백 2칸)해 이어 쓴다.
- **불릿 사용 기준**: 서로 독립적인 변경 사항이 2개 이상이면 `- ` 불릿으로 나눈다.
  하나의 흐름으로 이어지는 설명이면 불릿 없이 문단으로 쓴다.
- **섹션 참조**: 스펙 문서의 특정 절을 고치는 커밋이면 `§7`, `§3 D5`처럼 절 번호를
  본문에 명시해 어떤 부분이 바뀌었는지 바로 찾을 수 있게 한다.

## 6. 좋은 예시 vs 나쁜 예시

### 좋은 예시

1. `feat: 검토 루브릭·에스컬레이션 사다리 추가`
   — type과 대상이 분명하고, 명사형 종결에 가운뎃점으로 항목을 나열했다.
2. `docs: README 추가 — 설치·사용법·스모크 체크리스트`
   — 무엇을 추가했는지(README)와 그 안에 담긴 핵심 내용(상세 보충)이 함께 드러난다.
3. `docs: 스펙 §7 감독 메커니즘 정정 — 서브에이전트 제어 제약 반영`
   — 어느 절(§7)을 왜(제어 제약 반영) 고쳤는지 제목만으로 알 수 있다.
4. `fix: 워커 파견 조건 오류 수정 — 최대 3명 제한 누락 보완`
   — 무엇이 잘못됐고 어떻게 고쳤는지가 명확하다.

### 나쁜 예시

1. `docs: 수정`
   — 무엇을 수정했는지 대상이 없어 diff를 열기 전까지 아무 정보도 주지 못한다.
2. `feat: Add review rubric.`
   — 영어로 작성됐고 마침표까지 붙어 이 리포의 형식·언어 규칙을 둘 다 어긴다.
3. `docs: README를 추가했습니다.`
   — 구어체 종결(`~했습니다`)과 마침표를 사용해 명사형 종결·마침표 금지 규칙을 어긴다.
4. `fix: 여러가지 버그 수정 및 문서 정리 및 스킬 추가`
   — 서로 다른 type(`fix`+`docs`+`feat`)에 해당하는 변경을 한 커밋에 몰아넣어
     "및"으로 나열했다. type을 하나로 특정할 수 없으면 커밋을 나눠야 한다.

## 7. 커밋 전 셀프 점검 체크리스트

- [ ] 제목이 `type: 한국어 요약` 형식이고, 필요시 ` — 상세`가 붙어 있다.
- [ ] type이 §3 표에서 가장 가까운 것 하나로 특정된다(여러 type이 섞이지 않는다).
- [ ] 제목이 72자를 넘지 않고, 명사형 종결이며 마침표가 없다.
- [ ] 제목만 읽어도 "무엇이 바뀌었는지" 알 수 있다(대상이 빠져 있지 않다).
- [ ] 여러 변경을 나열할 때 가운뎃점(`·`)을 쓰고, 영어 어순 직역이 없다.
- [ ] 본문이 필요한 커밋이면 "왜"를 설명하고 있고, 단순 diff 재진술에 그치지 않는다.
- [ ] 본문 줄바꿈이 72~80자 선에서 이루어져 있다.
- [ ] 스펙 절을 고쳤다면 본문에 절 번호(`§n`)를 명시했다.
- [ ] 도구가 자동으로 붙인 `Co-Authored-By`/`Claude-Session` 푸터를 임의로 지우지 않았다.
- [ ] 전체가 한국어로 작성돼 있다(코드/식별자 등 불가피한 영어 제외).
