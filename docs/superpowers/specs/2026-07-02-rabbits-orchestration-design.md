# rabbits — 자율 서브에이전트 오케스트레이션 플러그인 (설계 스펙)

- 작성일: 2026-07-02
- 상태: 승인됨 — §7 감독 메커니즘 정정 반영 (구현 계획 writing-plans 진행)
- 대상 플랫폼: Claude Code 플러그인

---

## 1. 목적

메인 세션이 "대장(오케스트레이터)"이 되어, 사용자가 던진 작업을 스스로 **기획**하고,
필요한 **전문가 워커(서브에이전트)를 즉석 편성**해 일을 분배하고, 결과를 **검토·피드백**하며,
완료조건 충족까지 **완전 자율**로 몰고 가 최종 산출물을 보고하는 Claude Code 플러그인.

한 줄: **"대장토끼가 작업 받으면 → 팀 짜고 → 파견하고 → 감독·검토·재작업시켜 → 끝내고 보고한다."**

---

## 2. 핵심 개념 & 용어

- **대장 (Lead / Orchestrator)**: 지금 사용자와 대화 중인 메인 Claude 세션. `orchestrate` 스킬을 로드해
  기획·편성·파견·감독·검토·피드백·보고를 총괄한다. 유일하게 Agent 툴로 서브에이전트를 낳을 수 있는 주체.
- **워커토끼 (Worker)**: 대장이 즉석 디스패치하는 일회용 서브에이전트. 별도 파일로 저장되지 않으며
  세션 종료 시 사라진다. `claude` / `general-purpose` / `Explore` / `Plan` 등 내장 에이전트 타입에
  "전문가 역할 통째로 박은 프롬프트"를 얹어 전문성을 부여한다.
- **아키타입 (Archetype)**: 재사용 역할 프롬프트 템플릿(리서처·구현가·리뷰어·테스터·기획가). 대장이
  슬롯을 채워 워커 프롬프트를 조립한다. 맞는 게 없으면 같은 계약을 따르는 커스텀 역할을 즉석 작성.
- **감독 / 워치독 (Watchdog)**: 대장이 워커별 제한시간을 부여하고, 그 시간이 지나면 상태를 점검해
  "잘 진행 중"인지 "정체(멈춤)"인지 판정하고 개입하는 월클럭 감시 루프.
- **에스컬레이션 사다리 (Escalation Ladder)**: 워커가 완료조건에 못 미칠 때 전술을 바꿔가며 목표를
  계속 추격하는 절차(코칭 → 재편성·모델상향 → 분해 → 상한상향 재도전).

## 3. 확정된 설계 결정

| ID | 결정 | 근거 |
|----|------|------|
| D1 | 워커는 **즉석 일회용**(프롬프트 기반, 파일 저장 X) | 세션 중 새 에이전트 파일은 그 세션에서 디스패치 불가(§4). 프롬프트가 유일한 세션 내 동적 전문화 수단 |
| D2 | 실행은 **완전 자율**(중간 사용자 게이트 없음) | 사용자 선택. 진행상황은 나레이션으로 보여주되 멈추지 않음 |
| D3 | 엔진 = **스킬 + 아키타입 라이브러리**(엔진 B) | 코칭 루프(SendMessage)에 자연스럽고, 아키타입이 "전문성" 컨셉을 실현 |
| D4 | 피드백 = **goal-driven 에스컬레이션 사다리**(포기 없음) | 사용자 선택("끝까지 물고 감"). 라운드 수는 포기 지점이 아니라 전술교체 신호 |
| D5 | **워치독 감독**(maxTurns 1차 가드 + 제한시간 점검 + 개입) | 개별 워커의 무한 매달림(월클럭 폭주)을 잡는 안전장치. 보장 경로는 §7 정정판 |
| D6 | `agents/` 폴더 없음 | D1의 직접 결과. 플러그인이 가볍고 핫리로드 문제를 원천 회피 |

## 4. Claude Code 하드 제약 (연구 근거)

공식 문서 확인 결과. 이 제약들이 아키텍처를 강제한다.

1. **에이전트 핫리로드 불가**: 세션 시작 시 로드된 서브에이전트만 디스패치 가능. 세션 중
   `.claude/agents/`에 새 파일을 써도 그 세션에서는 `subagent_type`으로 호출 불가(재시작 필요).
   → "파일 저장형 워커"는 세션 내 즉시 사용이 원천 차단. **D1의 근거.**
2. **디스패치 시 커스텀 시스템프롬프트 주입 불가**: Agent 툴은 `subagent_type`(이름) + `prompt`(작업)만
   받음. 전문화는 오직 `prompt`에 역할을 통째로 담는 방식으로만 가능. → **D3 워커 전략의 근거.**
3. **서브에이전트 중첩 가능**(v2.1.172+): 워커도 자기 서브에이전트를 낳을 수 있음(메인에서 깊이 5까지).
   기본은 대장→워커 1단계. 필요 시 워커가 하위 워커를 두는 것 허용하되 남용 금지.
4. **플러그인 에이전트 제약**: 플러그인이 정의하는 에이전트는 hooks/mcpServers/permissionMode 미지원,
   Agent 툴 스폰은 메인 스레드에서만. → 대장은 플러그인 에이전트가 아니라 **메인 세션 + 스킬**이어야 함.
5. **슬래시 인자**: 스킬 본문에서 `$ARGUMENTS`로 자유 텍스트 인자 수신. → `/rabbits:run <작업>`.
6. **포그라운드 디스패치는 블로킹**: 대장이 포그라운드로 워커를 부르면 끝날 때까지 제어를 못 받음.
   감독하려면 `run_in_background` 필수. → **D5 워치독의 전제.**

근거 문서: code.claude.com/docs/en/{sub-agents, plugins, plugins-reference, skills, commands}.md

## 5. 아키텍처

```
User ──/rabbits:run <작업>──▶ [메인 세션 = 대장 (orchestrate 스킬 로드)]
       ┌───────────────────────────────────────────────────────────────┐
       │ 1.기획   작업 분해 → 미니 DAG(병렬/순차) + 워커별 완료조건         │
       │ 2.편성   아키타입 조합/커스텀 → 워커 프롬프트 + 모델·제한시간 배정  │ ← "서브에이전트 만들기"
       │ 3.파견   Agent 디스패치 (독립=병렬, 의존=순차, 장기=background)     │
       │ 3.5 감독 제한시간 워치독: 점검 → 진행중/정체 판정 → 개입           │
       │ 4.검토   루브릭 대조 → PASS / REVISE / ESCALATE                   │
       │ 5.피드백 SendMessage 재작업 → 안되면 에스컬레이션 사다리           │
       │ 6.보고   통합 산출물 + 리포트                                     │
       └───────────────────────────────────────────────────────────────┘
                                    │ Agent(claude/general-purpose/Explore/Plan, model=..)
                                    ▼
              [워커토끼들] 즉석 범용에이전트 + 아키타입 프롬프트 (파일 X, 리로드 X)
                          독립 워커는 한 메시지에 여러 Agent 호출로 병렬 파견
```

완전 자율: 1→6 사이 사용자 게이트 없음. 최종 보고와 §12의 좁은 안전밸브만 사용자에게 도달.

## 6. 6단계 프로토콜 상세

### 단계 0 — 인테이크
- `$ARGUMENTS`(작업) 파싱. 비었으면 작업만 사용자에게 요청(자율의 유일한 필수 입력).
- 목표 + 성공 정의를 1~2줄로 확정.

### 단계 1 — 기획 (PLAN)
- 작업을 서브작업으로 분해. 서브작업 간 의존관계로 **미니 DAG**(병렬/순차) 구성.
- 서브작업마다 정의: 목표 · **완료조건(acceptance criteria)** · 아키타입 · 모델티어 · 입력 · 출력계약.
- **팀 로스터/기획서**(간결 표)를 나레이션으로 제시(자율이므로 게이트 아님, 가시성 목적).
- 과분해 금지: 최소 인원 원칙(3명이면 되는 걸 10명 만들지 말 것).

### 단계 2 — 편성 (CAST) = "서브에이전트 만들기"
- 서브작업별로 워커 프롬프트 조립 = 아키타입 템플릿(§8) + 주입 항목:
  역할 타이틀 · 컨텍스트 · 정확한 작업 · 제약 · 출력 형식 · 완료조건 · "잡담 말고 결과만 반환".
- `subagent_type` 배정(보통 `claude`/`general-purpose`, 읽기전용 조사=`Explore`, 순수 기획=`Plan`).
- **모델티어** 배정: 잡일=haiku, 기본=sonnet, 어려움·최종통합=opus.
- **제한시간 T**(§7) + `maxTurns` 캡 배정.

### 단계 3 — 파견 (DISPATCH)
- 독립 워커 → **한 메시지에 여러 Agent 호출로 병렬** 파견.
- 의존 워커 → 상류 산출물이 나온 뒤 그 결과를 입력으로 순차 파견.
- **감독 대상(장기·불확실·무거움) = `run_in_background: true`**, 짧은 잡일 = 포그라운드(그냥 블로킹).
- 각 워커의 `agentId` 확보(완료 후 SendMessage 재작업 코칭용 — 실행 중 넛지는 하니스 의존, §7).

### 단계 3.5 — 감독 (WATCHDOG) — §7 상세
- 파견과 검토 사이를 가로지르는 감시 루프: 완료알림(자동)이 정상경로, 알림 없이 제한시간 T 경과 시
  ScheduleWakeup으로 깨어 점검 → 연장 or 개입(교체 워커 병렬 투입).

### 단계 4 — 검토 (REVIEW)
- 워커가 반환/완료하면 그 워커의 **완료조건** + `review-rubric.md`로 평가.
- 판정: **PASS**(전부 충족) / **REVISE**(고칠 수 있는 미달) / **ESCALATE**(근본적).
- 고위험 산출물은 구현가와 **독립된 리뷰어 워커**를 별도 파견해 교차검증(확증편향 감소).

### 단계 5 — 피드백 (COACH) & 에스컬레이션
- REVISE → `SendMessage(worker, 구체적 피드백)`으로 같은 워커가 컨텍스트 유지한 채 재작업 → 재검토.
- §9의 **에스컬레이션 사다리**로 목표까지 추격. 포기 없음, runaway 가드까지 지속.

### 단계 6 — 통합·보고 (SYNTHESIZE & REPORT)
- 모든 워커 PASS(또는 runaway 가드 소진) 후 대장이 산출물을 **통합**해 최종 결과물 생성.
- 최종 리포트: 한 일 요약 · 워커별(역할/결과/라운드) · 미해결 항목(있으면 이유 명시) · 통합 결과.

## 7. 감독(워치독) 메커니즘 & 3중 방어

### 감독 루프 (정정판 — 서브에이전트 실제 제약 반영)
```
편성 때 워커별: maxTurns(하드 자가종료 캡) + 제한시간 T(ScheduleWakeup 타이머) 부여
파견: 감독대상 = background(run_in_background) → 대장 고삐 유지, 폴링 불필요
   │
   ├─ 완료알림(자동 도착) ──────▶ 검토(4)로   [정상경로, 대부분]
   │
   └─ 완료알림 없이 T 경과(웨이크업) ──▶ 대장 점검:
        ├ (상태툴 되는 하니스면) TaskList/TaskGet로 생존 확인
        ├ 아직 도는 중 → "더 도네", T 연장 + 웨이크업 재장전
        │                (maxTurns가 결국 강제종료 → 무한 아님)
        └ 정체 의심 → 개입:
             ├ (되면) TaskStop 강제종료 후 재편성
             └ 안 되면(서브에이전트 kill 불확실) → 교체 워커 병렬 파견,
                원본은 maxTurns로 자멸하게 둠(대기 안 함) → 에스컬레이션 사다리
```

### 왜 이렇게 (서브에이전트 감시의 실제 제약 — 2026-07-02 확인, 정직하게)
연구 확인 결과 background **서브에이전트**는 Bash 백그라운드 태스크와 달리 경량 제어가 하니스 의존적:
- **완료 신호는 확실**: 워커가 끝나면 대장은 **자동 완료알림**을 받음(폴링 불필요).
- **상태폴링/강제kill은 불확실**: 서브에이전트용 TaskGet/TaskList/TaskStop 지원이 하니스마다 다를 수
  있음. 실행 중 워커에 SendMessage 넛지가 먹는지도 불확실(멈춘 뒤에만 될 수 있음).
- **트랜스크립트 정독 금지**: 워커 속을 다 읽으면 대장 컨텍스트 폭발(하니스 경고). 점검은 얕게.

→ "제한시간 되면 살펴본다"는 **ScheduleWakeup 타이머 + 완료알림 유무**로 실현. "멈춤 대응"의
  **보장 경로 = maxTurns 자가종료 + 교체 워커 병렬 투입**. 강제 kill/넛지는 되는 하니스에서만
  최적화로 사용. 어느 경우든 감독은 성립하고 폭주는 막힌다.

### 3중 방어 (폭주 차단) — 정정판
1. **`maxTurns` 캡 (1차 가드, 항상 통함)** — 워커가 루프/정체해도 N턴이면 강제 자가종료 → 완료알림
   → 검토가 부실 결과를 잡아 에스컬레이션. 서브에이전트에 **보장되는** 핵심 가드.
2. **ScheduleWakeup 제한시간 (월클럭 인지)** — 대장이 T에 깨어 완료 여부 점검 → 연장 or (되면) 강제종료.
3. **총량 runaway 가드 (런 단위)** — 런 전체 워커수/시도/토큰 상한. 소진 시에만 최선결과+미해결 보고(§9).

## 8. 아키타입 라이브러리 (archetypes.md)

작게 유지(고재사용 5종). 각 템플릿 = 역할선언 + 책임 + 하드제약 + 출력계약. 대장이 슬롯을 채운다.

| 아키타입 | 기본 subagent_type | 역할 | 하드 제약 | 출력 계약 |
|---------|-------------------|------|----------|----------|
| 리서처   | `Explore` | 읽기전용 조사·탐색 | 파일 수정 금지 | 발견 테이블(file:line 근거) |
| 구현가   | `claude` | 스펙대로 코드 작성/수정 | 스코프 밖 변경 금지 | diff 요약 + 파일 목록 |
| 리뷰어   | `claude`(읽기전용 운용) | 결과 감사·독립검증 | 수정 금지, 판정만 | 심각도 태그 findings |
| 테스터   | `claude` | 테스트 작성/실행 | 프로덕션 코드 변경 금지 | pass/fail + 커버리지 |
| 기획가   | `Plan` | 서브도메인 분해/설계 | 구현 금지 | 단계별 계획 |

- 맞는 아키타입이 없으면: 같은 계약(역할·제약·출력)을 따르는 **커스텀 워커 프롬프트** 즉석 작성.
- 모든 워커 프롬프트 공통 규칙: 역할·컨텍스트·작업·제약·출력형식·완료조건 명시 + "결과만 반환".
- 코드 산출물은 사용자 표준 준수: **2칸 들여쓰기, 한국어 주석**.

## 9. 검토 루브릭 & 에스컬레이션 사다리 (review-rubric.md)

### 판정 루브릭
- 완료조건 전부 충족? 완전한가? 정확한가? 스코프 내인가? 리포 관례(2칸/한국어 주석) 준수?
- **PASS**: 전부 충족 → 통합 대상.
- **REVISE**: 고칠 수 있는 미달 → 구체 피드백(정확히 뭐가 빈지 + "좋은 결과"의 모습). 대신 고쳐주지 말 것.
- **ESCALATE**: 근본적 문제 → 사다리 진입.

### 에스컬레이션 사다리 (goal-driven, 포기 없음)
```
1. 코칭      같은 워커 SendMessage 재작업(~2라운드, 싸고 컨텍스트 유지)
   └ 안되면 ▼
2. 재편성    새 워커 + 프롬프트 재작성(실패에서 학습) + 모델 상향(sonnet→opus)
   └ 안되면 ▼
3. 분해      서브작업 더 잘게 / 언블록 워커 투입(리서처로 정보 확보)
   └ 안되면 ▼
4. 상한상향  runaway 가드 상한을 높이고 메타전략 교체(각도 전환·opus·재분해) 반복
   └ 그래도 ▼ (가드 최종 소진 시에만)
5. 종료      최선결과 + "이건 이 이유로 미해결" 명시해 보고 (사용자에게 안 물음)
```
- "2라운드"는 1단계 안쪽의 **전술교체 신호**일 뿐, 목표 포기 지점이 아니다.

## 12. 에러처리 / 안전밸브

- **워커 사망/터미널 에러** → 1회 재편성, 안되면 리포트에 미해결 플래그.
- **정체(워치독)** → T 경과 점검 후 교체 워커 병렬 투입(되는 하니스에선 TaskStop 강제종료 후 재편성),
  원본은 maxTurns로 자가종료되게 둠(§7).
- **피드백 미달** → 에스컬레이션 사다리(§9).
- **무한루프** → 3중 방어(maxTurns / 워치독 / 총량 가드).
- **완전 자율에도 멈추는 좁은 예외(안전밸브)** — 이때만 사용자에게 표면화:
  1. 작업이 비었거나 **위험하게 모호**해서 안전한 가정이 불가.
  2. **파괴적·비가역** 동작 필요(삭제/외부발행/자격증명 등) — 하니스 안전규칙상 확인 필수.
  3. **진짜 외부 블로커**(없는 비밀번호, 사용자만 내릴 수 있는 결정 등 어떤 전략으로도 합성 불가).
  그 외 전부 자율 지속.

## 13. 테스트 / 검증 전략

프로토콜(스킬)이라 유닛테스트가 아님. 로컬 설치 후 대표작업으로 흐름 관찰 = 스모크 테스트.

- **T1 리서치형**: "이 코드베이스에서 X 어떻게 동작하는지 조사" → 리서처 워커 병렬 파견 → 통합 보고 확인.
- **T2 소코딩**: 작은 버그 수정 → 리서처→구현가→리뷰어 순차 DAG → 검토 루프 확인.
- **T3 다단계**: 독립 서브작업 3개 → 병렬 파견 + 감독 + 통합 확인.
- **T4 피드백 발동**: 일부러 덜 명세한 서브작업 → REVISE→코칭 루프가 실제로 도는지 확인.
- **T5 감독 발동**: 짧은 제한시간 부여한 background 워커 → 워치독 점검·개입 경로 확인.
- README에 위 체크리스트 수록.
