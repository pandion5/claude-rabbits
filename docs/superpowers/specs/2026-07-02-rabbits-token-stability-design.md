## 1. 목적

rabbits v1 오케스트레이션에서 **토큰을 적게 쓰면서 더 안정적인 산출물**을 뽑기 위한
프로토콜 확장 3개. 2026-07-02 T4/T5 스모크 런에서 실측된 누수 3곳을 직접 겨냥한다.

| 실측 누수 (T4/T5 런) | 겨냥 개념 |
|----------------------|-----------|
| 워커들이 각자 `git log`·리포 구조를 처음부터 재탐색 (중복 결제) | 개념 2 — 컨텍스트 팩 |
| xref 워커가 110행 표 반환, 대장은 3건만 사용 (과잉 반환) | 개념 1·3 — 결과 블록 + 반환 예산 |
| 느슨한 완료조건이 REVISE 라운드를 낳음 (풀 재실행 1회분) | 개념 1 — self_check 구조화 |

세 개념은 맞물린다: **팩(2)** 이 워커의 발견량을 줄여 → 반환이 자연히 작아지고 →
**예산(3)** 이 결과 블록에 담기고 → **블록(1)** 만 대장이 소비한다.

## 2. 핵심 제약 (설계를 강제하는 사실)

- **Agent 툴에 `schema` 파라미터 없음** — Workflow의 `agent()`와 달리 rabbits 워커는 Agent 툴로
  파견되므로 JSON 스키마를 툴 레벨에서 강제할 수 없다. 따라서 "구조화 반환"은
  **프롬프트 계약 + 대장 읽기 규율**로 실현한다 (툴 강제 아님).
- **대장은 워커의 최종 메시지만 소비** (v1 철칙 4). 이 확장은 그 "최종 메시지" 안에
  파싱 가능한 고정 블록을 심어 규율을 강화한다.
- **문서 전용 변경** — SKILL.md·archetypes.md의 프로토콜 문구만 바꾼다. 실행 코드 없음.

## 4. 개념 1 — 구조화 반환 계약 (`rabbits-result` 블록)

워커는 자유롭게 사고·작업하되 **최종 메시지 끝에 고정 블록**을 붙인다. 대장은 그 블록만
읽고 앞의 산문은 무시한다.

### 4.1 블록 포맷

    ```rabbits-result
    outcome: DONE            # DONE | BLOCKED
    deliverable: <결과물 또는 위치>
    evidence:
      - <판정에 필요한 최소 근거 file:line / 수치>
    self_check:
      - <완료조건1>: ✓
      - <완료조건2>: ✗ <이유>
    notes: <있으면 1줄>
    ```

- `outcome: BLOCKED`이면 `deliverable`·`self_check` 대신 `blocker: <이유>` + `need: <필요한 것>`.
- 워커는 `outcome`에 PASS/FAIL 같은 **판정을 쓰지 않는다** (E2). `self_check`는 완료조건별
  자가 대조 결과일 뿐, 최종 판정은 대장이 단계 4에서 내린다.
- 블록 앞 산문은 선택 — 워커가 사고 과정을 적어도 되지만 대장 컨텍스트에는 안 들어간다.

### 4.2 대장 소비 규칙 (SKILL.md 단계 4)

- 대장은 `rabbits-result` 블록의 `self_check`·`evidence`로 PASS/REVISE/ESCALATE를 판정한다.
- 블록이 없거나 깨졌으면 → REVISE 1순위 사유("결과 블록 누락")로 SendMessage 반려.

## 5. 개념 2 — 컨텍스트 팩

대장이 단계 1(기획) 끝에 **공통 배경 팩**을 1회 조립해, 단계 2(편성)에서 모든 워커
프롬프트의 `{{컨텍스트}}` 슬롯에 주입한다.

### 5.1 팩 내용 가이드

- 작업 배경 2~3줄
- 관련 파일 경로 + 각 한 줄 요약 (대장이 이미 파악한 것)
- 확정 사실·제약 (재조사 불필요분)
- 말미 고정 지시: **"이 팩에 있는 정보는 재조사하지 말 것. 팩에 없고 네 작업에 필요한
  것만 조사하라."**

### 5.2 조립 주체 (E3)

- **기본**: 대장 인라인 조립 (저렴, 대부분 작업).
- **하이브리드**: 탐색량이 크거나 대장이 배경을 모를 때 → 스카우트 리서처 워커 1명을
  먼저 파견, 그 `rabbits-result` 블록이 곧 팩이 된다.

---

## 6. 개념 3 — 반환 예산

archetypes 공통 규칙에 추가한다.

- **판정/결론 먼저, `evidence`는 최소** — 전체 목록·원문 덤프 금지. 대장이 더 필요하면
  SendMessage로 추가 요청한다.
- **소프트 상한(목표치, 하드 컷 아님)**: 잡일 ≤15줄 · 기본 ≤40줄 · 무거움 ≤80줄
  (`rabbits-result` 블록 기준). 근거상 초과가 불가피하면 초과하되 이유를 `notes`에 1줄.

`rabbits-result` 블록이 곧 예산 그릇이다 — 블록이 작으면 반환도 작다.

## 8. 비목표 (YAGNI)

- 툴 레벨 스키마 강제 — Agent 툴이 지원 안 함(§2). 프롬프트 계약으로 충분.
- 하드 컷 반환 제한 — 근거 누락 위험(E4).
- 엄격 필드 계약 + 자동 반려(포크 B) — REVISE 라운드 비용, 이번엔 채택 안 함.
- 교차검증(adversarial verify) 확대 — 토큰을 늘리므로 이 스펙의 목적(토큰↓)과 반대.
  v1의 "고위험만 독립 리뷰어" 유지.

---

## 9. 성공 기준

- 워커 프롬프트가 `rabbits-result` 블록을 요구하고, 대장이 그 블록만으로 판정 가능.
- 단계 1에서 컨텍스트 팩이 조립되고 단계 2에서 `{{컨텍스트}}`로 주입됨이 프로토콜에 명시.
- 공통 규칙에 반환 예산 3항이 존재하고 각 아키타입이 블록 반환을 참조.
- 다음 실제 rabbits 런에서 워커 간 중복 재탐색이 사라지고, 대장이 소비하는 반환물이
  블록 단위로 축소됨(정성 관찰).
