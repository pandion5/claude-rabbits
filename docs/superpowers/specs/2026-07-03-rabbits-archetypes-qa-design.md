## 4. 신규 아키타입 5종 (`archetypes.md` 9종 → 14종)

전부 공통 규칙 상속, 기존과 같은 구조(역할선언·책임·하드제약·출력계약). 템플릿 펜스는
탑레벨(들여쓰기 0), 말미 고정 문구는 기존 9종과 글자 단위 동일.

### 4.1 요약표

| # | 아키타입 | 성격 | 기본 subagent_type / 모델 | 핵심 출력 (블록 매핑) |
|---|---------|------|--------------------------|----------------------|
| 10 | 통합가 (Integrator) | 여러 워커 산출물 병합·충돌 해소·최종 조립 | `claude` / sonnet, 무거움 opus | deliverable=통합 결과물 + 병합 결정 목록, evidence=충돌 지점별 해소 근거 |
| 11 | 디버거 (Debugger) | 재현→최소화→가설→계측→수정 루프 | `claude` / sonnet, 난해 opus | deliverable=근본원인 + 수정 요약, evidence=재현 절차·계측 결과, self_check에 "수정 후 미재현" 포함 |
| 12 | 검증가 (Verifier) | 산출물을 **실제 실행**해 동작 관찰(E2E) | `claude` / sonnet | deliverable=시나리오별 관찰 결과, evidence=실행 명령 + 출력 발췌 |
| 13 | 문서가 (Doc Writer) | 문서 작성·개정 전문, `{{독자}}` 슬롯 | `claude` / sonnet, 잡일 haiku | deliverable=문서 위치 + 구성 요약, evidence=핵심 구성 결정 근거 |
| 14 | 마이그레이터 (Migrator) | 반복 패턴 일괄 변환(rename·API 교체·포맷 통일) | `claude` / sonnet, 잡일 haiku | deliverable=변환 규칙 + 파일 수 집계, evidence=잔존 검색 0 증거 |

### 4.2 아키타입별 하드 제약

- **통합가**: 통합 대상 산출물만 취급 — 신규 기능 추가 금지. 병합 결정(채택/기각/수정)마다
  근거 1줄. 충돌을 임의 뭉개지 말고 해소 불가면 BLOCKED.
- **디버거**: **근본원인 확정 전 수정 금지**(증상 패치 금지). 재현 불가면 추측 수정 대신
  BLOCKED. 수정 후 원 재현 절차로 미재현 확인 의무.
- **검증가**: 관찰만 — 어떤 파일도 수정 금지. **관찰하지 않은 것을 통과로 보고하지 않는다**
  (실행 못 한 시나리오는 "미실행"으로 명시).
- **문서가**: 문서 파일만 수정(코드 변경 금지). `{{독자}}`(대상 독자)에 맞는 깊이·용어 선택.
- **마이그레이터**: 변환 규칙 밖 리팩토링 금지. **전수성 의무** — 대상 전수 조사 → 변환 →
  잔존 0 확인(검색 증거)까지가 한 사이클.

### 4.3 경계 명문화 (archetypes.md에 기재)

- **검증가 vs 테스터**: 테스터 = 테스트 코드를 작성·실행 / 검증가 = 완성 산출물의
  실제 플로우를 구동해 관찰. 코드 산출물 없는 검증은 검증가.
- **디버거 vs 구현가**: 구현가 = 스펙대로 작성 / 디버거 = 원인 규명 후 최소 수정.
  "고쳐줘"형 서브작업에 구현가를 쓰면 증상 패치 위험 — 버그면 디버거.

## 5. QA팀 (`teams.md` 5번째 팀)

### 5.1 QA리스트 규약 (영속 산출물)

- 위치: **작업 대상 리포**의 `.rabbits/qa-checklist.md`. git 추적(회귀 기준은 영속이어야 의미).
  없으면 QA팀 첫 투입 시 생성.
- 내용 = **검증 시나리오 원장**: 기능별 섹션(`## 기능명`), 항목 = `- 시나리오 — 기대 결과`.
- **순회 결과(✓/✗)는 리스트에 기록하지 않는다**(A6). 결과는 그 런의 최종 리포트에.
- **리스트 파일 갱신은 대장이 수행** — QA 설계 워커(테스터)는 시나리오를 결과 블록으로
  반환만 한다(테스터 하드제약 "테스트 파일 외 수정 금지"와 충돌 방지). 대장이 반환된
  시나리오를 리스트에 반영한 뒤 순회를 파견한다. 삭제는 기능 제거 시에만.

### 5.2 편성·흐름

| 역할 | 아키타입 | 임무 |
|------|---------|------|
| QA 설계 | 테스터 1 (QA 브리프) | 신규 기능의 검증 시나리오 작성 → 결과 블록으로 반환(리스트 반영은 대장) |
| 순회 실행 | 검증가 1~N | 리스트 **전량 순회**(신규+기존) 실제 실행·관찰. 규모 적응(A7) |
| 실패 대응 | 디버거 0~N | 실패 항목 근본원인 규명·수정 — 발견 시만 |

흐름: 테스터 시나리오 반환 → 대장 리스트 반영 → 전량 순회 → 실패 항목 디버거 수정 →
해당 항목 재검증.
수정 코드는 고위험 산출물 — 단계 4 규칙대로 독립 리뷰어 교차검증(보안팀 스캔→패치→재스캔과
같은 패턴).

### 5.3 발동 신호 (대장 자동판단 — T2 유지)

- 기능 구현이 포함된 런: 구현 산출물 PASS 후·통합(단계 6) 전에 투입.
- "QA 돌려줘"·회귀 점검 성격 요청.

### 5.4 판정 기준

- 신규 기능 항목이 리스트에 추가됐는가.
- 전량 순회가 실제 실행됐는가 — 항목별 실행 증거(명령+출력), "미실행" 항목 0.
- 실패 항목이 수정·재검증 통과했는가.

## 8. 비목표 (YAGNI)

- **기존 팀 4종 편성 재검토** — 무변경(A2).
- **QA리스트에 순회 결과 기록** — 부패 유발(A6). 결과는 런 리포트에.
- **QA 자동 스케줄링·훅 연동** — 발동은 대장 자동판단만(T2 유지).
- **테스트 프레임워크 강제** — 검증가는 실행 관찰이지 특정 프레임워크 의존 아님.
- **검증된 코드 패턴 저장소** — 별도 개념, 사용자 기각(2026-07-03).
- **review-rubric.md·SKILL.md 단계 4 변경** — T7 유지.
