## 1. 목적

반복되는 전문 검토·조사 작업을 **팀 단위 프리셋**으로 상비한다. 대장이 작업 성격을 보고
자동으로 팀을 투입해, 매 런마다 전문 워커 편성을 즉석에서 재발명하는 비용을 없앤다.

| 팀 | 전문 분야 |
|----|----------|
| 테크팀 | 기술적 가능성 검토 (실현 가능성·리스크·대안) |
| 법무팀 | 라이선스·상업적 사용 가능성 검토 |
| 보안팀 | 취약점 점검 + 발견 시 패치 구현 + 재스캔 검증 |
| 서치팀 | 웹+코드 멀티모달 검색·자료 조달 (단독 임무 + 타팀 지원) |

---

## 2. 핵심 제약·원칙 (설계를 강제하는 사실)

- **팀 = 편성 레시피(문서 계약), 상주 조직 아님.** v1의 "즉석 일회용 워커" 철학 유지 —
  팀 프리셋은 아키타입 조합 + 흐름 + 판정 기준을 미리 적어둔 것일 뿐, 워커는 여전히
  Agent 툴로 즉석 편성된다.
- **발동은 대장 자동판단만.** 별도 진입점(`/rabbits:security` 등)·사용자 게이트 없음.
  안전밸브 3조건(SKILL.md)은 그대로 적용된다.
- **문서 전용 변경.** 프로토콜 문서만 바꾼다. 실행 코드 없음.
- **기존 가드 전부 상속.** 워커 상한 12(→18), 최소 인원 원칙, 코칭 2라운드,
  에스컬레이션 사다리, 공통 규칙(`rabbits-result` 블록·반환 예산·컨텍스트 팩 존중).
- **SKILL.md 상주 토큰 최소화.** 팀 발동 신호 요약(팀당 1줄)만 SKILL.md에 상주,
  상세(편성·흐름·판정)는 teams.md로 분리해 팀 매칭 시에만 Read한다 —
  archetypes.md·review-rubric.md와 같은 지연 로드 패턴.

## 4. 신규 아키타입 4종 (`archetypes.md` 5종 → 9종)

전부 공통 규칙(`rabbits-result` 블록, 반환 예산, 컨텍스트 팩 존중, BLOCKED 프로토콜)을
상속하고, 기존 5종과 같은 구조(역할선언·책임·하드제약·출력계약)를 따른다.

### 4.1 요약표

| # | 아키타입 | 성격 | 기본 subagent_type / 모델 | 핵심 출력 (deliverable/evidence 매핑) |
|---|---------|------|--------------------------|--------------------------------------|
| 6 | 테크 검토가 (Tech Assessor) | 읽기전용 기본, 스파이크 필요 시만 쓰기 허용 | `Explore` (스파이크 시 `claude`) / sonnet, 어려움 opus | 가능성 판정(가능/조건부/불가) + 근거 + 리스크 + 대안 |
| 7 | 라이선스 감사가 (License Auditor) | 읽기전용 | `Explore` / sonnet | 의존성별 라이선스 표 + 위험등급(안전/주의/위험) + 근거 조항 |
| 8 | 보안 감사가 (Security Auditor) | 읽기전용, **방어 전용** | `Explore` / sonnet | 발견별 위치·심각도(치명/중요/사소)·수정 제안 |
| 9 | 서처 (Searcher) | 읽기전용, 웹+코드 멀티모달 | `claude`(WebSearch·WebFetch 필요) / sonnet, 잡일 haiku | 발견 + 출처(URL/file:line) + 확신도 |

### 4.2 아키타입별 하드 제약

- **테크 검토가**: 판정에는 반드시 근거(문서·코드·실험 결과) 첨부. 스파이크 코드는
  검증 목적의 최소량만, 산출물로 남기지 않음(스코프 밖 파일 변경 금지 유지).
- **라이선스 감사가**: 모든 보고서 말미에 **"이 검토는 법률 자문이 아니다"** 고지 필수.
  라이선스 판정은 원문 조항 근거로만 — 통념·추측 금지.
- **보안 감사가**: **공격 코드·익스플로잇 작성 금지** 명시. 발견은 심각도순 정렬.
  비밀값을 발견해도 결과 블록에 원문을 옮기지 않는다(위치만 보고).
- **서처**: 조달(retrieval) 전문 — 특정 자료·원문·문서를 빨리 찾아 출처와 함께 반환.
  분석·해석은 최소화(그건 리서처 몫).

### 4.3 리서처 vs 서처 경계 (archetypes.md에 명문화)

- **리서처** = 깊은 조사·분석. 질문에 답을 만든다.
- **서처** = 넓은 조달·소싱. 자료를 찾아온다(웹 원문, CVE 항목, 라이선스 전문, 문서 링크).

## 5. `teams.md` — 팀 프리셋 4종 (신설)

각 팀 항목 = **발동 신호 · 편성표(아키타입 조합 + 기본 인원 + 모델) · 팀 흐름 · 판정 기준**.
파일 서두에 공통 원칙 명시: 인원은 기본값(대장이 규모 조절, T6), 워커 상한·최소 인원
원칙 적용, 팀 워커도 일반 워커와 동일하게 단계 3~5(파견·감독·검토·코칭)를 탄다.

### 5.1 테크팀

- **발동 신호**: "기술적으로 가능한가?" 판단, 기술 스택 선택, 아키텍처 결정, 신기술 도입.
- **편성**: 테크 검토가 1~3 (2명 이상이면 독립 각도 분담 — 예: 성능·호환성·구현복잡도)
  + 서처 0~1 (기술 문서·벤치마크 조달 필요 시).
- **흐름**: 병렬 검토 → 대장이 판정 종합. 검토가 간 결론 충돌 시 3번째 검토가 투입
  또는 opus 상향(사다리 2단계 준용).
- **판정 기준**: 판정(가능/조건부/불가)에 근거 있는가, 리스크와 대안이 실행 가능한 수준인가.

### 5.2 법무팀

- **발동 신호**: 라이선스 검토, 상업적 사용 가능성, 의존성 도입 판단, 배포·판매 전 점검.
- **편성**: 라이선스 감사가 1 + 서처 0~1 (라이선스 원문·판례 조달 필요 시).
- **흐름**: 의존성 인벤토리 → 라이선스 식별 → 호환성·상업 사용 판정 → 보고서.
- **판정 기준**: 의존성 커버리지 전수인가, 위험등급마다 근거 조항 있는가,
  "법률 자문 아님" 고지 있는가.

### 5.3 보안팀

- **발동 신호**: 보안 점검 요청, 인증·입력처리·비밀 취급 코드 작업, 배포 전 점검.
- **편성**: 보안 감사가(스캐너) 1~2 + 구현가(대응가 = 기존 구현가 아키타입 + 패치 브리프)
  0~N — **발견이 나왔을 때만** 투입.
- **흐름**: 스캔 → 심각도순 발견 목록 → 치명·중요 발견을 대응가가 패치 →
  **스캐너 재스캔으로 검증**. 패치는 고위험 산출물 → 기존 단계 4 규칙대로 독립 리뷰어
  교차검증.
- **판정 기준**: 스캔 커버리지(인젝션·하드코드 비밀·의존성 CVE·설정 미스 4범주),
  패치가 재스캔을 통과했는가, 새 취약점을 만들지 않았는가.

### 5.4 서치팀

- **발동 신호**: 조사·자료 조달이 작업의 핵심일 때. 또는 타팀·타워커가 외부 자료를
  필요로 할 때(지원 파견).
- **편성**: 서처 1~N (웹/코드 모달별 병렬 — 멀티모달 스윕).
- **흐름**: 모달별 병렬 스윕 → 대장 종합. 지원 파견 시 결과 블록이 해당 서브작업의
  `{{컨텍스트}}` 보강분이 된다.
- **이중 역할**: (a) 단독 임무 (b) 타팀 지원. 단계 1의 컨텍스트 팩 스카우트
  (상위 스펙 E3 하이브리드)도 서처로 수행 가능 — 스카우트 규칙과 자연 결합.
- **판정 기준**: 출처 명시(URL/file:line) 전수인가, 확신도 표기 있는가, 조달물이
  요청 범위를 커버하는가.

## 6. `SKILL.md` 통합

- **단계 1 (PLAN)**: 팀 발동 신호 요약표(팀당 1줄) 추가. 서브작업 분해 시 작업 성격이
  신호에 걸리면 해당 서브작업을 팀 단위로 편성 계획하고, 로스터 표에 **"팀" 컬럼**을
  추가해 소속을 표기한다(무소속 워커는 공란).
- **단계 2 (CAST)**: 팀 매칭된 서브작업이 있으면 이 스킬의 베이스 디렉토리의 `teams.md`를
  Read한다(이 단계 첫 진입 시 1회 — archetypes.md와 동일 패턴). 편성표대로 아키타입 +
  팀 브리프(흐름·판정 기준 요약)를 슬롯에 주입해 워커를 조립한다.
- **단계 3 (DISPATCH)**: 팀 소속 워커 라벨 = `rabbits:팀/역할: 작업 요약`
  (예: `rabbits:보안팀/스캐너: 인증 모듈 취약점 스캔`). 무소속 워커는 기존
  `rabbits:역할: 작업 요약` 유지.
- **단계 4 (REVIEW)**: 변경 최소 — 공통 루브릭 그대로, 팀 산출물은 teams.md의 팀 판정
  기준을 완료조건 해석의 보조로 쓴다(T7).

## 8. 비목표 (YAGNI)

- **별도 스킬 진입점** (`/rabbits:security` 등) — 대장 자동판단으로 충분(T2).
- **선제적 런타임 방어 코드 구축** (rate limit·보안 헤더 등 방어 인프라 신규 구현) —
  보안팀은 점검+패치까지(T3). 방어 인프라가 필요하면 그건 일반 구현 작업으로 다룬다.
- **공격 도구·익스플로잇 작성** — 방어 전용. 아키타입 하드 제약으로 명시(§4.2).
- **팀 간 직접 통신·상주 팀 상태** — 모든 조율은 대장 경유(v1 철칙 유지).
- **review-rubric.md 개정** — 공통 루브릭 유지(T7).
- **아키타입별 전용 판정 루브릭** — teams.md 판정 기준으로 충분.
